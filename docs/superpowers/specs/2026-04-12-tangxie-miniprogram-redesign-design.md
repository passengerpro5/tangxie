# 糖蟹小程序改版设计方案

## 1. 目标

本次改版只针对糖蟹小程序用户侧页面，不包含后台。

改版目标不是增加新功能，而是在不改变现有核心能力边界的前提下，完成一轮统一的 UI/UX 重设计：

1. 让产品形成一致的品牌气质。
2. 让首页、事项页、排期页和任务详情页的视觉语言统一。
3. 让信息层级更清楚，用户更容易快速理解“现在应该看什么”。
4. 在移动端保持轻量、克制、可读，不为了设计感牺牲工作区。

## 2. 本轮结论

### 2.1 品牌方向

确认采用：

`温和治愈 + 效率优先`

具体含义：

1. 外观上更柔和、低刺激、愿意每天打开。
2. 骨架上仍然是效率工具，强调结构清晰、状态明确、操作成本低。
3. 不做强人格化 AI 助手界面，也不做冷硬后台式工具感。

### 2.2 范围

本轮只覆盖以下页面：

1. 日程首页
2. 事项页
3. 排期页
4. 任务详情页

不包含：

1. 后台页面
2. 用户侧新增配置入口
3. 用户侧新增日志入口
4. 任何新的业务能力扩展

## 3. 不可突破的约束

这些约束已经确认，后续实现不得偏离：

1. 首页必须仍然是 `日程`。
2. `日程 / 事项 / 排期` 三个现有 tab 的功能定义不改。
3. 允许调整样式、布局、层级、文案和状态表达，但不允许偷偷改成功能重构。
4. `今日节奏` 只允许出现在 `日程` 首页。
5. `事项` 和 `排期` 页面不显示 `今日节奏` 模块。
6. 任务详情页只做信息展示重设计，不新增操作能力。
7. 多级 tab 在保证交互清晰的前提下，尽量不跨行，尽量减少顶部高度占用。

## 4. 信息架构

### 4.1 一级页面结构

用户侧继续保留三页：

1. `日程`
2. `事项`
3. `排期`

这组三页本质上是页面级主导航，而不是页面内部的普通切换器。

### 4.2 页面关系

1. `日程` 是主首页，承担“今天怎么看、今天做什么”的入口角色。
2. `事项` 承担任务看板视角。
3. `排期` 承担规划 / 甘特式时间安排视角。
4. `任务详情` 是从上述页面进入的独立详情页，只负责更完整地展示现有任务信息。

## 5. 导航系统

### 5.1 一级导航

一级导航采用已经确认的 `浮岛混合` 方案。

表现原则：

1. `日程 / 事项 / 排期` 作为高层级浮岛导航存在。
2. 视觉权重要高于内容模块，用户一眼就能知道自己在什么页面。
3. 保留温和圆角和轻浮起感，但不能厚重到像横幅或头图。

### 5.2 二级与三级导航

二级模式切换与三级时间粒度切换采用压缩式横向控制带。

原则：

1. 二级和三级尽量合并到同一水平区域。
2. 能不换行就不换行。
3. 文案命名要短，避免把控制带挤爆。
4. 视觉上弱于一级导航，但仍然要明确可切换。

### 5.3 顶部高度控制

本轮导航系统的核心要求之一，是减少顶部导航对工作区的侵占。

因此：

1. 不保留额外情绪化头图或独立彩条。
2. 页面顶部直接从主导航开始。
3. 只有 `日程` 页在主导航和控制带之后，再出现轻量 `今日节奏` 摘要。

## 6. 视觉系统

### 6.1 色彩

基础色系采用暖米白、浅杏和低刺激暖灰。

强调色采用：

1. 杏橙
2. 珊瑚橙
3. 柔和粉橙

用途：

1. 一级选中态
2. 重点任务块
3. 节奏摘要和关键提醒

正文不使用纯黑，而采用深茶棕，降低压迫感。

### 6.2 形态

整体形态原则：

1. 圆角偏柔和，但不做过度奶油风。
2. 卡片边界清楚，层次靠浅边框、轻投影和材质感完成。
3. 页面不追求大面积装饰，而是通过细腻的表面层级体现品质。

### 6.3 动效

动效只服务于理解，不服务于炫技。

原则：

1. 页面进入要轻。
2. 切换反馈要明确。
3. 拖动、选中、展开态要清楚。
4. 小程序运行环境内避免重动画和高性能开销。

## 7. 页面设计

### 7.1 日程首页

#### 角色

日程首页仍是整个小程序的主入口。

它要同时满足两件事：

1. 让用户快速知道今天整体节奏。
2. 让用户尽快进入时间轴主工作区。

#### 顶部结构

顶部顺序固定为：

1. 一级主导航：`日程 / 事项 / 排期`
2. 紧凑控制带：日程页内部模式 + 时间粒度
3. 轻量折叠式 `今日节奏`

#### 今日节奏

`今日节奏` 必须满足以下设计要求：

1. 默认只占一行。
2. 默认只显示少量关键指标。
3. 支持点击展开和收起。
4. 展开后高度仍然不能超过工作区域约三分之一。
5. 它是辅助信息，不得压过时间轴主体。

#### 主工作区

时间轴仍然是主内容，不改能力。

优化方向：

1. 降低无效噪声。
2. 当前日和重点任务更容易识别。
3. 时间块更温暖、更容易扫读。
4. 编辑、拖拽、冲突等状态表达更清晰。

### 7.2 事项页

#### 角色

事项页继续承担看板视图，不改变看板能力边界。

#### 顶部结构

事项页顶部只保留：

1. 一级主导航
2. 一条紧凑控制带

不显示 `今日节奏`。

#### 优化方向

1. 让列头信息更清楚。
2. 让列之间的推进关系更清楚。
3. 让用户更容易判断当前最该关注哪一列、哪张卡片。
4. 卡片层级比原先更明确，但不改成别的产品形态。

### 7.3 排期页

#### 角色

排期页继续承担规划 / 甘特式视图。

#### 顶部结构

排期页顶部同样只保留：

1. 一级主导航
2. 一条紧凑控制带

不显示 `今日节奏`。

#### 优化方向

1. 让阶段关系更清楚。
2. 让时间跨度更容易理解。
3. 降低“工程图表感”，提升“计划视图感”。
4. 继续保留原有排期功能和语义。

### 7.4 任务详情页

#### 角色

任务详情页这轮不升级为操作页，只升级为更完整的展示页。

#### 第一屏信息

第一屏优先展示：

1. 任务标题
2. 当前状态
3. 时间信息
4. 优先级
5. 分类或来源标签

#### 后续信息层

中下部继续展示：

1. 计划说明
2. 相关背景或来源信息
3. 历史记录

#### 明确禁止

本轮任务详情页禁止新增以下内容：

1. 固定操作带
2. 标记完成
3. 重新安排
4. 编辑信息
5. 其他新增操作能力

如果这些能力未来要做，必须作为单独功能改版重新定义，不可在本轮 UI 改版中偷偷带入。

## 8. 状态设计

### 8.1 日程页状态

重点状态包括：

1. 正常查看态
2. 选中态
3. 编辑态
4. 冲突态
5. 空状态

设计要求：

1. 正常态安静，不喧宾夺主。
2. 编辑态和冲突态反馈必须明显。
3. 空状态不只是“没有内容”，而要给出下一步理解线索。

### 8.2 事项页状态

重点状态包括：

1. 默认看板态
2. 高优先级筛选态
3. 空列状态

要求用户能一眼分辨：

1. 哪列处于压力较高状态
2. 哪类任务最值得优先处理

### 8.3 排期页状态

重点状态包括：

1. 不同时间粒度
2. 规划密集状态
3. 信息不足或跨度较长状态

要求是：

1. 视图切换清楚
2. 内容密时不乱
3. 时间跨度大时仍然可读

## 9. 文案语气

统一文案语气为：

1. 克制
2. 温和
3. 清楚
4. 不命令式

产品表达要像“可靠地帮用户整理”，而不是“系统在发指令”。

## 10. 本轮不做

明确排除以下内容：

1. 后台改版落地
2. 用户侧配置页
3. 用户侧日志页
4. 任务详情新增操作
5. `日程 / 事项 / 排期` 的功能改写
6. 复杂的 AI 人格化包装
7. 为追求高级感而增加多余高度和装饰

## 11. 实现约束

实现阶段应遵守以下要求：

1. 小程序端坚持移动优先和大触控区。
2. 继续使用现有小程序技术栈和页面能力。
3. 视觉更新应尽量建立在现有页面和组件结构上，不无故发明一套平行系统。
4. 样式实现优先保证稳定、可维护和运行性能。

## 12. 验收标准

如果改版实现正确，最终应满足：

1. 用户一眼能看出 `日程 / 事项 / 排期` 是主导航。
2. `今日节奏` 只在 `日程` 首页出现。
3. `事项` 和 `排期` 顶部明显更短。
4. 多级 tab 基本不跨行，顶部占高明显下降。
5. 日程首页仍然像日程首页，没有被做成别的产品。
6. 任务详情信息更完整，但没有越界成操作页。
7. 整体品牌气质统一为“温和治愈 + 效率优先”。

## 13. 原型状态

本设计方案对应的确认版高保真原型已完成，小程序范围内包含：

1. 日程首页
2. 事项页
3. 排期页
4. 任务详情页

当前原型的作用是确定方向和约束，不直接代表最终像素级实现细节。实现阶段应以本 spec 为准，把确认过的交互和视觉原则落进真实页面。
