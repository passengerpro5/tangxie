# 糖蟹小程序 1.0 设计方案

## 1. 项目目标

糖蟹 1.0 是一款面向个人用户的任务自动排期小程序。

产品核心不是通用待办，也不是团队协作工具，而是帮助用户把零散输入的任务内容整理成可执行的时间安排，并在合适的时间进行小程序内提醒。

1.0 聚焦以下闭环：

1. 用户输入任务内容，支持文本、图片、文档。
2. 系统抽取任务信息，缺失时通过对话追问补全。
3. 系统基于 deadline、任务语义和紧急程度给出优先级初排。
4. 用户确认优先级和关键约束后，系统生成推荐排期。
5. 首页以日程和任务看板两种方式展示排期结果。
6. 系统按排期结果进行小程序内提醒。

## 2. 非目标

1.0 不包含以下能力：

1. 多人协作、任务分派、团队权限。
2. 复杂项目依赖和甘特式项目管理。
3. 短信、邮箱、企业微信等多渠道推送。
4. 全局自动重排全部任务。
5. 大量面向终端用户的高级排期规则自定义。

## 3. 核心产品定位

糖蟹 1.0 的定位是“个人任务排期助理”。

产品主入口是“安排任务”对话浮层，而不是传统任务列表编辑页。用户可以直接提交一段文本、一份文档或一张截图，由系统先做任务识别和信息补全，再把任务写入日程。首页承担的是结果展示与轻量调整，而不是从零创建任务的主工作区。

## 4. 核心用户流程

### 4.1 单任务流程

1. 用户进入首页，看到“日程”和“任务看板”。
2. 用户点击“安排任务”。
3. 用户输入文本，或上传图片、文档。
4. 系统抽取任务草稿。
5. 如果缺少 deadline、预计耗时或优先级判断依据，系统继续追问。
6. 信息足够后，系统生成推荐排期。
7. 用户确认后，任务写入日历，并生成提醒。

### 4.2 多任务流程

1. 用户一次输入多个任务，或上传包含多个任务的材料。
2. 系统抽取多条任务草稿。
3. 系统先基于 deadline 和紧急程度语义生成优先级初排。
4. 用户确认或修改排序。
5. 系统按优先级生成推荐排期。
6. 用户确认后，任务进入日历和看板。

## 5. 功能模块

### 5.1 任务输入模块

支持以下输入：

1. 文本输入。
2. 图片或截图上传。
3. 文档上传。

输出结果不是直接排期，而是结构化任务草稿，包括：

1. 任务标题。
2. 任务描述。
3. 来源类型。
4. 候选 deadline。
5. 候选耗时。
6. 抽取置信度与待确认字段。

### 5.2 AI 澄清模块

该模块负责判断任务是否具备排期条件。

若缺失关键信息，系统通过对话继续追问，重点围绕：

1. 什么时候必须完成。
2. 大概需要多久。
3. 相比现有任务的重要程度。

若为多任务场景，系统先给出基于 deadline 和语义的初步优先级，再要求用户确认。

### 5.3 智能排期模块

系统在信息充足后生成推荐排期。

1.0 不追求复杂优化算法，优先保证排期可解释、可确认、可调整。

基本能力包括：

1. 识别当前可用时间段。
2. 按优先级顺序占用时间。
3. 确保任务在 deadline 前完成。
4. 无法排下时给出明确提示和操作建议。

### 5.4 展示模块

展示分为两个视图：

1. 日程视图：查看任务具体落在哪一天、哪个时段。
2. 任务看板：查看任务状态和整体安排情况。

### 5.5 提醒模块

1.0 只做小程序内提醒。

提醒类型至少包括：

1. 任务开始前提醒。
2. deadline 临近提醒。
3. 当天任务汇总提醒。

## 6. 页面与交互结构

### 6.1 首页

首页顶部有两个切换：

1. 日程。
2. 任务看板。

#### 日程视图

1. 以时间轴方式展示任务时间块。
2. 支持点击查看详情。
3. 支持基础拖动或调整。
4. 右下角保留“+”浮动按钮。

#### 任务看板视图

建议 1.0 先采用四列状态：

1. 待补信息。
2. 待安排。
3. 已安排。
4. 已完成。

### 6.2 安排任务浮层

这是产品核心入口，采用底部上滑浮层。

建议包含：

1. 历史记录区。
2. 对话区。
3. 输入区。
4. 上传入口。
5. 安排按钮。

按钮状态要求：

1. 信息不足时，引导继续补充，不进入排期。
2. 信息足够时，才允许生成推荐排期。

### 6.3 任务详情页

详情页展示：

1. 任务内容。
2. 来源材料。
3. deadline。
4. 预计耗时。
5. 优先级。
6. 排期结果。
7. 提醒设置。

动作包括：

1. 标记完成。
2. 重新安排。
3. 编辑信息。
4. 查看提取记录。

### 6.4 今日任务区

1.0 可作为首页模块或轻量页面，用于展示：

1. 今日任务列表。
2. 即将开始任务。
3. 临近 deadline 的任务。

## 7. 数据模型

### 7.1 Task

任务核心实体。

建议字段：

1. `id`
2. `title`
3. `description`
4. `source_type`
5. `status`
6. `deadline_at`
7. `estimated_duration_minutes`
8. `priority_score`
9. `priority_rank`
10. `importance_reason`
11. `created_by_ai`
12. `user_confirmed`

### 7.2 TaskInputSource

记录任务输入来源。

建议字段：

1. `id`
2. `task_id`
3. `source_type`
4. `raw_text`
5. `file_url`
6. `file_name`
7. `parsed_summary`
8. `created_at`

### 7.3 ClarificationSession

记录一次信息补全过程。

建议字段：

1. `id`
2. `task_id`
3. `current_missing_fields`
4. `messages`
5. `ai_extracted_fields`
6. `user_confirmed_fields`
7. `session_status`
8. `provider_id`
9. `model_name`
10. `prompt_version`

### 7.4 ScheduleBlock

记录任务占用的时间块。

建议字段：

1. `id`
2. `task_id`
3. `start_at`
4. `end_at`
5. `block_type`
6. `schedule_status`
7. `created_by`

一个 Task 未来可以对应多个 ScheduleBlock。1.0 默认可以先按一个任务一个时间块落地，但结构不能锁死。

### 7.5 Reminder

记录提醒计划与状态。

建议字段：

1. `id`
2. `task_id`
3. `schedule_block_id`
4. `remind_at`
5. `reminder_type`
6. `delivery_status`
7. `read_status`

### 7.6 UserSchedulePreference

记录用户的基础排期偏好。

建议字段：

1. `user_id`
2. `workday_start`
3. `workday_end`
4. `lunch_break`
5. `default_focus_duration`
6. `timezone`

### 7.7 AIProviderConfig

记录 AI 服务商配置。

建议字段：

1. `id`
2. `name`
3. `provider_type`
4. `base_url`
5. `api_key_encrypted`
6. `default_model`
7. `enabled`
8. `created_at`
9. `updated_at`

说明：

1. `api_key` 不应明文保存，至少需要加密存储。
2. `provider_type` 用于区分 OpenAI 兼容协议或其他未来扩展协议。

### 7.8 AIModelBinding

记录不同业务场景绑定的模型配置。

建议字段：

1. `id`
2. `provider_id`
3. `scene`
4. `model_name`
5. `temperature`
6. `max_tokens`
7. `timeout_seconds`
8. `enabled`
9. `is_default`

### 7.9 PromptTemplate

记录提示词模板和版本。

建议字段：

1. `id`
2. `scene`
3. `template_name`
4. `system_prompt`
5. `developer_prompt`
6. `version`
7. `is_active`
8. `created_at`
9. `updated_at`

说明：

1. 同一业务场景应允许存在多个版本。
2. 线上调用必须能追踪到具体模板版本。

## 8. 排期规则

### 8.1 排期前置条件

若缺少以下任一关键字段，则任务不可进入正式排期：

1. deadline。
2. 预计耗时。

优先级信息可以由系统初步推断，但必须允许用户确认或修正。

### 8.2 优先级初排

优先级分数建议由以下部分组成：

1. deadline 紧迫度。
2. 文本中的紧急程度语义。
3. 用户确认后的顺序修正。

### 8.3 时间占用规则

推荐的 1.0 规则：

1. 先排高优先级任务。
2. 优先排入最早可执行空闲时段。
3. 时间块必须晚于当前时间。
4. 时间块必须早于 deadline。
5. 尽量使用连续时间段。

### 8.4 排不下时的处理

若任务在 deadline 前找不到足够时间：

1. 不静默失败。
2. 不擅自全局重排其他任务。
3. 明确提示用户当前无法按时完成。
4. 提供可选动作，例如提高优先级、延后其他任务、拆分任务、修改 deadline。

### 8.5 用户调整优先级

用户在日历或详情页中的手动调整优先级高于 AI 推荐结果。后续提醒和状态更新应基于用户确认后的最终排期。

## 9. 技术架构

建议采用五层结构：

1. 小程序前端。
2. 业务后端。
3. AI Provider 网关层。
4. 任务理解与排期编排层。
5. 提醒调度层。

### 9.1 小程序前端

负责：

1. 首页日程和看板展示。
2. 安排任务浮层对话。
3. 任务详情页。
4. 今日任务和提醒查看。

### 9.2 业务后端

负责：

1. 用户、任务、附件、会话、排期、提醒数据管理。
2. 对外提供标准 API。
3. 权限与配置分层。

建议 API 范围：

1. 创建任务。
2. 上传附件。
3. 继续澄清对话。
4. 生成推荐排期。
5. 确认排期。
6. 查询今日任务。
7. 标记完成。

### 9.3 AI Provider 网关层

该层必须支持手动接入 AiHubMix 等 API 中转服务商，推荐兼容 OpenAI 风格接口。

必须支持配置：

1. `base_url`
2. `api_key`
3. `model`
4. `temperature`
5. `max_tokens`
6. `timeout_seconds`
7. 启用状态

该层职责是屏蔽不同服务商差异，向上层提供统一的 AI 调用接口。

### 9.4 任务理解与排期编排层

该层不直接依赖某家模型，而是基于“使用场景”调用 AI Provider 网关。

建议拆分场景：

1. `task_extract`
2. `clarification`
3. `priority_rank`
4. `schedule_generate`
5. `reminder_copy`

该层负责：

1. 提取任务草稿。
2. 识别缺失字段。
3. 生成追问内容。
4. 生成优先级初排。
5. 生成推荐排期结果。

### 9.5 提醒调度层

负责根据最终确认的 `ScheduleBlock` 和 `Reminder` 执行提醒任务。

1.0 只做小程序内提醒，但调度逻辑需要独立于排期逻辑。

## 10. 管理后台设计

1.0 直接纳入后台管理界面，不采用手工改数据库的方式。

### 10.1 AI 服务商管理

管理员可：

1. 新增服务商。
2. 编辑服务商。
3. 启用或停用服务商。
4. 配置 `base_url`、`api_key`、协议类型、默认模型。
5. 进行连通性测试。

### 10.2 模型配置管理

管理员可为不同场景绑定不同模型，例如：

1. 任务提取模型。
2. 追问模型。
3. 优先级判断模型。
4. 排期生成模型。
5. 提醒文案模型。

每个场景至少需要支持配置：

1. 所属服务商。
2. 模型名。
3. temperature。
4. max_tokens。
5. 超时时间。
6. 是否为默认配置。

### 10.3 提示词模板管理

管理员可：

1. 在线编辑内置提示词。
2. 为不同使用场景配置不同提示词模板。
3. 启用指定版本。
4. 回滚到旧版本。

提示词至少需要版本化，便于追踪某次排期所使用的模板。

### 10.4 调试与日志

后台建议支持：

1. 连通性测试。
2. 最近调用日志查看。
3. 调用所使用的 provider、model、prompt version 查看。
4. 响应摘要查看。

建议额外记录：

1. 请求场景。
2. 请求耗时。
3. 是否命中降级或 fallback。
4. 最终是否写入正式排期。

## 11. 工程约束

为保证后续可演进性，建议明确以下约束：

1. 业务层不得写死具体模型名。
2. AI 调用必须按场景路由，不按固定模型路由。
3. 提示词模板必须版本化。
4. `api_key` 必须加密存储，不在日志中明文输出。
5. 提醒调度不得直接依赖未确认的任务草稿。
6. 用户手动确认后的排期结果优先于 AI 推荐结果。

## 12. MVP 范围

### 12.1 用户端 MVP

1. 文本、图片、文档输入任务。
2. AI 抽取任务信息。
3. AI 追问缺失字段。
4. AI 给出优先级初排。
5. 用户确认后生成推荐排期。
6. 首页展示日程和任务看板。
7. 查看任务详情和标记完成。
8. 小程序内提醒。

### 12.2 管理端 MVP

1. AI 服务商配置。
2. 模型配置。
3. 提示词模板编辑和切换。
4. 调用测试与日志查看。

## 13. 1.1 预留方向

1. 一个任务拆分成多个时间块。
2. 未完成任务自动重排。
3. 更细的用户工作时间偏好设置。
4. 更丰富的外部提醒渠道。
5. 排期解释能力。

## 14. 开发顺序建议

建议按以下顺序推进：

1. 先完成核心数据模型和后端骨架。
2. 跑通“安排任务浮层”的最小闭环。
3. 完成推荐排期生成和确认流程。
4. 完成首页日程和看板展示。
5. 接入图片和文档解析。
6. 接入小程序内提醒。
7. 完成后台管理界面。

## 15. 建议结论

糖蟹 1.0 应作为一个双端系统推进：

1. C 端小程序负责任务输入、补全、排期展示和提醒接收。
2. B 端管理后台负责 AI 服务商、模型和提示词的配置与调试。

底层以 `Task -> ClarificationSession -> ScheduleBlock -> Reminder` 作为核心业务链路。AI 部分通过可配置 Provider 网关接入 AiHubMix 等服务，不将 API 地址、模型名和提示词写死在业务代码中。
