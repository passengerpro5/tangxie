# 糖蟹每日复盘与成绩单设计方案

## 1. 目标

本次设计要在糖蟹里新增一套围绕 `每日复盘` 和 `今日成绩单` 的用户侧机制。

目标不是做一套新的评分审判系统，而是用更适合分享的方式，刺激用户在完成一天工作后：

1. 更愿意把自己的成果发给微信好友或群聊。
2. 更愿意形成每天收工复盘的习惯。
3. 在分享和复盘过程中持续获得情绪价值。
4. 让糖蟹成为“帮用户把今天收好”的产品，而不是“再给用户一次压力”的产品。

次级目标：

1. 为后续的自然邀请和口碑传播埋下入口。
2. 为未来周报、成长身份卡和称号养成系统打基础。

## 2. 本轮结论

### 2.1 产品定位

本轮确认采用：

`收工解锁型每日成绩单`

具体含义：

1. 用户完成当天最后一个任务后，糖蟹发起一轮轻量收工提醒。
2. 用户通过事实确认式复盘，把今天完成了什么、没完成什么、接下来怎么处理确认清楚。
3. 完成复盘后，系统解锁一张适合分享的 `今日成绩单`。
4. 成绩单不展示分数，不展示等级，而是用 `有梗称号 + 高光指标 + 一句总结` 来完成分享激励。

### 2.2 核心价值观

本轮机制的价值观已经确认：

`结果优先，努力兜底`

具体解释：

1. 系统优先奖励真实完成和真实推进。
2. 用户即使没有清空全部任务，只要认真复盘、诚实更新进度、妥善处理剩余任务，依然可以拿到体面且值得分享的结果。
3. 这套系统奖励的是“把今天收好”，不是“把今天完美做完”。

### 2.3 分享气质

本轮分享内容确认采用：

`战报型表达，但不做分数和等级`

也就是说：

1. 这张卡片仍然像“今天交出了一张不错的战报”。
2. 但它不出现冰冷分数，也不出现用户容易产生抗拒的等级标签。
3. 第一眼吸引分享的是 `称号`，第二眼提供可信度的是 `高光标签和指标`。

### 2.4 分享场景

分享出口确认采用：

`同时支持微信好友 / 群聊 + 朋友圈`

但主设计重心按以下顺序处理：

1. 先优先适配 `微信好友 / 群聊`
2. 再兼容 `朋友圈`

因此成绩单内容更像：

`顺手发给熟人的有趣收工战报`

而不是：

`正式公开发布的成绩海报`

## 3. 范围

### 3.1 本轮包含

本轮只覆盖糖蟹小程序用户侧与每日收工相关的新增能力：

1. 完成当天最后一个任务后的收工提醒
2. 每日复盘入口和复盘流程
3. 已完成任务确认交互
4. 未完成任务进度确认和重新安排流程
5. 重新安排确认页或确认层
6. `今日成绩单` 页面
7. 成绩单分享内容和分享出口
8. 首发称号系统和文案策略

### 3.2 本轮不包含

本轮明确不做：

1. 后台管理或运营后台能力
2. 社区、公开榜单、好友排行榜
3. 强邀请裂变页、邀请码机制或明显拉新海报
4. 周报、月报或成长人格总览页
5. 独立成就体系、勋章墙或积分商城
6. 重新定义现有任务、日程、事项、排期的主功能边界

## 4. 不可突破的约束

这些约束已经确认，后续设计和实现不得偏离：

1. 不显示 `分数`
2. 不显示 `战报等级`
3. 每日成绩单必须在 `复盘完成后` 才解锁
4. 复盘必须是 `事实确认式复盘`，不能变成写作文或情绪问卷
5. 分享卡片的品牌露出必须很轻，不做强拉新
6. 视觉气质必须延续现有小程序的 `温和治愈 + 效率优先`
7. 交互必须移动优先、低负担、可在收工时快速完成
8. 有未完成任务时，不能直接跳过处理逻辑去解锁成绩单
9. 如果用户在复盘里修改了安排，必须先和用户确认新的安排结果
10. 这套系统必须让用户感觉“被理解和被奖励”，不能像第二次绩效考核

## 5. 北极星与衡量

### 5.1 北极星行为

本轮最重要的目标行为是：

`用户完成每日复盘后，主动打开并分享今日成绩单`

### 5.2 关键指标

建议实现后重点关注：

1. `收工提醒打开率`
2. `每日复盘完成率`
3. `成绩单解锁率`
4. `成绩单分享点击率`
5. `成绩单真实分享率`
6. `次日回访率`

### 5.3 次级观察指标

可以辅助观察：

1. 未完成任务的重排确认率
2. 连续收工天数分布
3. 不同称号风格的分享点击差异
4. 成绩单查看后返回小程序的后续留存

## 6. 用户心智

本轮机制要建立的用户心智不是：

`我又被系统打了一次分`

而是：

`我把今天交代清楚了，糖蟹给了我一个很懂我的收工结果`

用户最终要获得三层感受：

1. `确认感`
   今天做了什么、还剩什么，不再糊着结束一天。
2. `体面感`
   即使没全部做完，只要残局收好了，也依然是体面收工。
3. `分享感`
   这张卡片有梗、有信息量、又不会过于羞耻，值得发给别人看。

## 7. 主流程

### 7.1 触发时机

主触发时机确认采用：

`用户完成当天最后一个任务后，自动提示进入每日复盘`

这里的“最后一个任务”在第一期按糖蟹当天计划内的可完成任务计算，不要求引入复杂预测。

### 7.2 主链路

主链路固定为：

1. 用户完成当天最后一个任务
2. 系统弹出轻量收工提醒
3. 用户进入每日复盘
4. 用户确认今天已完成的任务
5. 如果仍有未完成任务，系统追问当前进度和是否需要重新安排
6. 如果有新的安排改动，系统先展示新的安排方案并请求确认
7. 所有事实确认完成后，解锁 `今日成绩单`
8. 用户查看成绩单并选择是否分享

### 7.3 流程原则

这条链路必须满足：

1. 每一步都围绕“确认事实”而不是“做额外思考”
2. 收工节奏要顺，不允许连续出现重表单
3. 如果用户没有未完成任务，流程要明显更短
4. 成绩单必须像奖励，而不是必须完成的作业

## 8. 页面与状态设计

### 8.1 收工提醒

#### 角色

在用户完成当天最后一个任务后，作为进入复盘的第一层提示。

#### 目标

1. 让用户知道：今天可以顺手收个工。
2. 让用户感觉：这不是额外负担，而是马上能解锁一个结果。

#### 表达原则

1. 文案轻，不命令
2. 不直接强调“复盘”，而强调“收工”或“解锁今日成绩单”
3. 明确告诉用户：这是一个很短的确认流程

#### 建议动作

1. `去收工`
2. `稍后再说`

### 8.2 每日复盘页

#### 角色

作为用户确认今日事实的主页面。

#### 第一屏结构

第一屏优先展示：

1. 页面标题：例如 `把今天收一下`
2. 今日任务列表
3. 已完成任务确认区
4. 底部确认按钮

#### 核心交互

用户先看到系统列出的今日任务，逐项点选哪些已经完成。

设计要求：

1. 每个任务卡片一目了然
2. 点选完成时，任务卡片出现局部庆祝动效
3. 动效要克制、局部、轻盈，不能像游戏爆炸特效
4. 已确认完成的任务状态必须清楚可回退

#### 局部庆祝动效建议

只允许使用轻量级表达：

1. 卡片角标亮起
2. 小范围粒子或闪点
3. 标题区域短促上浮
4. 柔和颜色扩散

不允许：

1. 全屏遮罩动画
2. 夸张音效依赖
3. 阻断用户继续操作的重反馈

### 8.3 未完成任务处理

#### 触发条件

当复盘页中仍存在未完成任务时，系统自动进入未完成任务处理流程。

#### 目标

1. 不让未完成任务悬而不决
2. 让用户带着清楚的结果收工

#### 处理内容

对于每个未完成任务，系统至少要确认：

1. 当前进度如何
2. 是否需要重新安排

第一期不做复杂文本填写，优先使用结构化选择：

1. `还没开始`
2. `推进了一点`
3. `基本完成，差收尾`

再进一步确认：

1. `移到明天`
2. `改到本周其他时间`
3. `暂时挂起`
4. `保持原样`

### 8.4 重新安排确认

#### 触发条件

如果用户在未完成任务处理流程中产生了新的安排改动，必须进入确认环节。

#### 目标

1. 避免用户在收工时不知不觉改掉安排
2. 让“尾巴已处理”这件事有明确落点

#### 展示原则

1. 用简洁摘要展示改动结果
2. 强调“改了什么”，而不是重做完整排期
3. 支持回退修改

#### 建议动作

1. `确认新安排`
2. `返回修改`

### 8.5 成绩单解锁页

#### 角色

作为每日复盘的奖励页和分享页。

#### 核心任务

1. 让用户感到今天已经被好好命名
2. 让用户愿意顺手分享
3. 让用户看到今天的推进、稳定感和收尾成果

#### 解锁条件

只有当以下条件满足时才解锁：

1. 已完成任务已被确认
2. 所有未完成任务都已更新当前状态
3. 若有新安排改动，用户已确认

## 9. 成绩单内容结构

### 9.1 顶部结构

成绩单顶部不展示分数和等级，而采用：

`有梗称号 + 高光标签串`

例如：

`今日截止线驯兽师`

`推进 3h · 连续收工 5 天 · 尾巴已收好`

### 9.2 三个高光指标

本轮确认固定使用以下三个指标：

1. `推进时长`
2. `连续收工天数`
3. `收尾力`

#### 推进时长

作用：

1. 体现今天确实有投入
2. 避免卡片只剩“完成了几个任务”这种单一结果

第一期定义建议：

1. 以今日已确认完成任务的累计推进时长为主
2. 对未完成但已推进的任务，可在结构化进度选择中补充少量已推进信息
3. 不要求第一期做精细计时器，优先保证可信和可解释

#### 连续收工天数

作用：

1. 体现稳定感
2. 形成每日回访动力

定义建议：

1. 用户连续完成每日复盘并解锁成绩单，即累计一天
2. 中断则重新计算

#### 收尾力

作用：

1. 体现糖蟹区别于普通任务产品的独特价值
2. 强调“今天不是只看做完了多少，也看你有没有把尾巴收住”

表达原则：

1. 不展示抽象数值
2. 使用结果型短语展示

例如：

1. `尾巴已收好`
2. `残局也交代清楚了`
3. `明天的路已经留好了`

### 9.3 总结文案

在高光指标下方展示一段很短的总结文案。

要求：

1. 先对今天的结果做肯定
2. 再补一句理解式表达
3. 语气轻，不鸡汤，不端着

例如：

1. `今天推进得很稳，连剩下的尾巴也处理干净了。`
2. `不是所有任务都清空了，但你把今天交代得很漂亮。`
3. `你不只是做完了一些事，也替明天把路理顺了。`

### 9.4 底部动作

建议动作：

1. `发给好友 / 群聊`
2. `发朋友圈`
3. `返回修改复盘`

品牌露出原则：

1. 只做轻量 logo 或轻量尾巴
2. 不额外加入明确拉新话术
3. 让卡片更像用户自己的战报，而不是糖蟹的广告

## 10. 称号系统

### 10.1 基本结论

称号是本系统最重要的分享钩子。

因此本轮不做单一风格称号，而是直接建立：

`至少 120 个首发称号`

### 10.2 四个语气池

称号必须同时覆盖四种风格：

1. `轻松嘴贫型`
2. `可爱反差型`
3. `热血游戏型`
4. `温柔会夸型`

四个语气池全部保留，不能只做一种。

### 10.3 行为场景池

建议首发至少覆盖以下行为场景：

1. `高完成`
2. `高推进`
3. `高收尾力`
4. `连续收工`
5. `逆风补救`
6. `稳扎稳打`
7. `极限冲刺`
8. `温和恢复`

### 10.4 组合方式

建议采用：

`行为场景池 × 语气池`

例如：

1. `高推进 × 轻松嘴贫型`
2. `连续收工 × 温柔会夸型`
3. `逆风补救 × 可爱反差型`

按这个方式组织称号库，首发很容易达到 120+，同时也便于后续扩容。

### 10.5 称号示例

#### 轻松嘴贫型

1. `今日截止线驯兽师`
2. `拖延税精准避税员`
3. `任务清仓临门一脚`

#### 可爱反差型

1. `没摆烂也算超常发挥`
2. `差点散架但还是交卷了`
3. `今日居然有在发光`

#### 热血游戏型

1. `任务清剿先锋`
2. `本日推进暴击手`
3. `收官连击发动者`

#### 温柔会夸型

1. `今天把生活稳住的人`
2. `有在认真接住自己`
3. `替明天留好了路的人`

### 10.6 称号命中原则

称号不能随机乱给，而要遵守：

1. 先根据当天行为信号判断用户属于哪个主要场景
2. 再从对应语气池中挑选一个最近没出现过的称号
3. 避免短期重复
4. 避免在明显高推进的日子命中过于“惨兮兮”的称号
5. 避免在明显低完成但高收尾的日子命中过于炫耀型称号

## 11. 内部判定逻辑

虽然用户看不到分数和等级，但系统内部仍然需要一套轻量信号来支持称号和文案生成。

第一期建议内部只判断，不展示：

1. `完成情况`
2. `推进情况`
3. `收尾情况`
4. `连续情况`
5. `补救情况`

这些内部信号的作用是：

1. 决定更适合哪个行为场景池
2. 决定总结文案的角度
3. 决定高光标签串中的第三段描述

必须注意：

1. 内部逻辑可以复杂一些
2. 对用户呈现必须简单、轻盈、可分享

## 12. 视觉与交互原则

### 12.1 视觉方向

沿用当前小程序已经建立的方向：

`温和治愈 + 效率优先`

因此成绩单和复盘页必须：

1. 保留暖纸感和低刺激配色
2. 维持深茶棕文字和柔和强调色
3. 用局部材质感和轻层级提升品质
4. 不做强竞技感、排行榜感或游戏大厅感

### 12.2 动效方向

本轮允许强调 `微交互`，但必须克制。

适合做动效的点：

1. 任务卡点选完成
2. 复盘确认后解锁成绩单
3. 分享前卡片轻微上浮或显现

不适合做动效的点：

1. 未完成任务处理全流程
2. 大面积页面切换
3. 长时间庆祝动画

### 12.3 交互负担控制

收工阶段是一天中用户耐心较低的时刻，因此：

1. 复盘流程要短
2. 每一步只做一件事
3. 文案要极清楚
4. 用户要随时能回退，但不能迷路

## 13. 文案语气

本轮统一文案语气为：

1. 有一点点机灵
2. 温和，不审判
3. 清楚，不含糊
4. 会夸人，但不过度肉麻

不能采用：

1. KPI 汇报腔
2. 绩效考核腔
3. 过强鸡汤腔
4. 过强中二腔

## 14. 分期建议

### 14.1 第一期

只落最核心链路：

1. 收工提醒
2. 每日复盘
3. 未完成任务处理
4. 重排确认
5. 今日成绩单
6. 首发称号系统
7. 分享出口

### 14.2 第二期

可以追加：

1. 称号偏好记忆
2. 更丰富的称号池
3. 分享卡模板变化
4. 连续收工的特殊彩蛋

### 14.3 第三期

再考虑扩展：

1. 周报
2. 成长身份卡
3. 轻量邀请引导

## 15. 验收标准

如果本设计实现正确，最终应满足：

1. 用户不会把它理解成一个打分系统
2. 用户能在很短时间内完成每日复盘
3. 已完成任务确认过程有明确但克制的奖励感
4. 未完成任务不会被糊弄过去，而是会被明确处理
5. 成绩单顶部以 `称号 + 高光标签串` 为核心，不出现分数和等级
6. 成绩单里的三个高光指标固定为 `推进时长 / 连续收工天数 / 收尾力`
7. 称号系统不是单一风格，首发至少具备 120 个可用称号
8. 分享卡整体更像用户自己的收工战报，而不是糖蟹广告
9. 整体视觉和交互不脱离当前小程序的品牌气质
10. 用户完成一次收工后，会觉得“今天被好好收住了”

## 16. 原型与实现关系
